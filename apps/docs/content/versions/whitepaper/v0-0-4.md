## Executive Summary

**EmMittr** is infrastructure for launching tokens with
participation-based economics. It introduces **"emmissions"** ---
derivative tokens representing a staked position in a fee-earning LP
pool. When you perform valuable actions, project tokens move from a
reserve into the pool, and you receive **em{Token}** --- a liquid claim
on that pool's growing value.

**The core primitive is Work:** any verifiable action that creates value
for the project. When you perform Work, you *emmitt* --- triggering the
reserve-to-pool transfer and receiving your emmission tokens. These
tokens appreciate as trading fees compound in the pool.

**The flywheel:** Launch token (1 ETH market cap) → Set aside emmission
reserve → Tie emmissions to Work → Work drives attention and activity →
Trading generates fees → 40% of fees compound in pool → Emmissions
appreciate → Early participants rewarded → More incentive to Work
**early** when the project needs it most.

**Work is pluggable:** For Emerge (content generation), Work = creating
content. For Moltbot skills, Work = invoking the skill. For games, Work
= minting cards. Any verifiable action can be wired to emmissions.

## What is an Emmission?

An **"emmission"** (written as em{TOKEN}) is a derivative token that
represents a staked position in a fee-earning LP pool. Here's the
mechanic:

-   **Staked backing:** When you emmitt, project tokens move from the
    reserve into the pool. Your emmission is backed by those staked
    tokens.

-   **Fee earnings:** 40% of all LP trading fees compound into the pool.
    As the pool grows, each emmission becomes worth more of the
    underlying project token.

-   **Liquid derivative:** After a 30-day vesting period, emmissions can
    be traded freely or burned to redeem the underlying value.

**Example:** Aviary launches the BIRD token. Users who perform valuable
actions (Work) earn **emBIRD**. Each emBIRD is backed by BIRD staked
from the reserve into the pool. As people trade BIRD, 40% of LP fees
flow into the emBIRD pool. Your emBIRD becomes worth more BIRD over
time.

*Think of it as: a staking derivative where the yield comes from LP
trading fees.* You earn it by participating, it appreciates from trading
activity, and you can sell it or burn it to redeem the underlying.

## Part I: The Opportunity

### App Tokens Hold Massive Potential

Application tokens represent a powerful mechanism for early-stage
project funding. They align incentives between builders and users,
bootstrap network effects, and create liquid markets for participation.
The innovation that Clanker popularized---sharing trading fees with
creators---proved that fee distribution creates sustainable economics.
But current implementations leave significant value on the table.

### The Participation Problem

Current app tokens lack incentives for participation in the project's
core mechanisms. Users buy tokens hoping for appreciation, but there's
no economic connection between using the product and earning from it.
The token floats separate from the value being created.

### The Friction Problem

Forcing users to pay in the project token for core functionality adds
friction. Users must acquire the token before experiencing value,
creating a chicken-and-egg problem. This friction reduces adoption,
which reduces fees, which reduces token value---a negative spiral.

### The Buy Pressure Problem

Sustainable buy pressure is hard to achieve. Most projects rely on
speculation or forced utility (pay-in-token), both of which create
friction or fade over time. Meanwhile, fees flow only to creators---but
fees are smaller than they could be because there's no mechanism to
share them with participants who would amplify the project.

### The EmMittr Solution

**EmMittr solves these problems simultaneously:** Early supporters
participate in fee earnings through emmissions. A staking mechanism
locks tokens in a pool, creating buy pressure without friction. Users
don't need to hold the token to use the product---they earn emmissions
by participating, which creates organic demand. The result is a flywheel
where participation drives fees, fees drive appreciation, and
appreciation rewards early participants.

## Part II: Implementation

EmMittr is built as a **native launchpad** from day one. A single
transaction deploys a token with full EmMittr mechanics: emmission
distribution, fee splitting, and permanent liquidity locking. The
off-chain SDK handles Work measurement and reporting.

### Smart Contract Architecture

The protocol consists of 8 core contracts organized into three layers:

### Launchpad Layer

Handles token creation and initial liquidity:

-   **EmMittrFactory:** Main entry point. Deploys token + all
    infrastructure in one transaction.

-   **EmMittrToken:** Native ERC-20 with built-in 2% fee-on-transfer.
    Fees route automatically to FeeSplitter.

-   **EmMittrLpLocker:** Locks Uniswap V3/V4 LP positions permanently.
    Rug-proof by design.

### Engine Layer

Manages emmissions and distribution:

-   **EmMittrEngine:** Main orchestrator per-token. Holds reserve,
    manages emmission pool, and exposes reportWork()---the single entry
    point for projects to report user actions. Only the project's
    authorized address can call it.

-   **EmToken:** Emmission ERC-20 (em{TOKEN}). Mintable only by Engine.
    30-day vesting lock on new mints.

-   **EmPool:** Emmission liquidity pool. Simple x\*y=k AMM where
    emmissions trade against base tokens.

### Infrastructure Layer

Shared utilities and math:

-   **FeeSplitter:** Routes all fees: 40% → EmPool, 50% → Creator, 10% →
    Protocol.

-   **DecayCurve:** Library implementing Tokens(n) = Base / (1 + K × n).
    Pure math, no state.

### Deployment Flow

When a creator launches through EmMittr:

1.  Creator calls EmMittrFactory.launch(name, symbol, reservePct,
    workConfig)

2.  Factory deploys EmMittrToken with fee-on-transfer mechanics

3.  Factory creates Uniswap pool with single-sided liquidity (1 ETH
    market cap)

4.  LP position locked permanently via EmMittrLpLocker

5.  Factory deploys EmMittrEngine, EmToken, EmPool for this token

6.  Reserve allocation (e.g., 5% of supply) transferred to Engine

7.  Project receives authorized address for calling reportWork() via SDK

*Result:* One transaction, fully operational token with emmissions, fee
distribution, and permanent liquidity.

## Part III: The EmMittr Model

EmMittr provides economic infrastructure that makes participation-based
token launches work. It solves three fundamental problems: how to lock
liquidity permanently, how to distribute value to participants fairly,
and how to create liquid markets for earned positions.

### The Flywheel: Meet Aviary

To illustrate how EmMittr works, imagine a company called **Aviary**
launching their app token **BIRD**. Users who take valuable actions in
Aviary's app earn **emBIRD**---the emmission token. Here's how the
flywheel works:

1.  **BIRD launches at 1 ETH market cap** (single-sided liquidity, no
    capital required from Aviary)

2.  **Reserve set aside:** 5% of BIRD supply is locked as the "emmission
    reserve" to back all future emBIRD

3.  **Work defined:** Aviary decides what actions count as valuable
    (create content, use features, refer friends)

4.  **Work triggers emmission:** User completes Work → BIRD moves from
    reserve to pool → user receives emBIRD equal to the amount moved,
    keeping the price stable

5.  **Trading generates fees:** Every BIRD trade has 2% fees

6.  **40/50/10 split:** 40% of fees compound in the emBIRD pool in BIRD,
    50% goes to Aviary, 10% to EmMittr in ETH

7.  **emBIRD appreciates:** As trading fees compound, emBIRD becomes
    worth more BIRD. Early participants earn more emBIRD per Work (decay
    curve)

**The key insight:** emBIRD is a staking derivative where the yield
comes from LP trading fees. Your emBIRD increases in value as more BIRD
enters the pool, effectively giving you a claim on future fees.

### Emmissions (em{Token})

The core primitive is the em{Token}---an emmission that represents a
staked position in the project's fee-earning pool. Key properties:

-   **Reserve-backed at mint:** When you emmitt, reserve tokens transfer
    to pool, keeping price stable at mint time

-   **30-day vesting lock:** Newly emmitted tokens are locked for 30
    days to prevent immediate dumping

-   **Tradeable after unlock:** Once unlocked, emmissions can be
    bought/sold on the EmMittr exchange

-   **Appreciates from fees:** 40% of all trading fees compound in the
    pool, increasing emmission value

### The Decay Curve

Emmissions are distributed according to a decay function that rewards
early participants:

### *Tokens(n) = Base / (1 + K × n)*

Each emmission produces fewer tokens than the last. The K parameter
controls steepness. At K = 0.002, the first Work earns approximately 3x
what later Work earns. This creates meaningful early-adopter advantage
without making late participation worthless.

**Key insight:** Creator fees are undiluted. No matter how many people
emmitt, the creator always gets 50% of trading fees (1% of volume)
directly. Emmissions dilute each other; creator revenue doesn't dilute.

## Part IV: Work — Measuring and Rewarding Value

Every project already measures what its users do. Stripe fires a webhook
when someone pays. Analytics track content creation. APIs log every
call. EmMittr's insight is simple: **these measurements are Work.** If a
project can tell us that a user did something valuable, we can turn it
into emmissions.

### The pattern is always the same:

### Work → Emmitt → Reserve → Pool → Value

There is no oracle. No on-chain verification. The project is the
authority on what counts as Work. They call *emmittr.reportWork(user,
action, amount)* and the protocol handles the rest---decay curve,
reserve transfer, emmission minting, vesting.

### The EmMittr SDK

The SDK is how projects measure and report Work. It's designed as a
ramp---dead simple to start, increasingly powerful as projects grow.

### Layer 1: Bring Your Own

Day one integration. Projects pipe their existing events to EmMittr. If
you have a webhook, a database event, or an API call, you have
everything you need.

-   **Stripe webhook fires** → SDK reports payment as Work → user
    emmitts

-   **API call logged** → SDK reports invocation as Work → user emmitts

-   **Content created** → SDK reports generation as Work → user emmitts

This is the on-ramp. Projects aren't building anything new---they're
sharing events they already have.

### Layer 2: Measurement Tools

Some of the most valuable Work is hard to measure. Was that social post
actually good? Did that referral lead to a real user? EmMittr provides
tooling for the measurements projects want but struggle to build:

-   **Social quality scoring:** Not just "did they post" but "was it
    meaningful engagement?" Measures reach, replies, sentiment---rewards
    quality over spam

-   **Commerce plugins:** Shopify, WooCommerce, and payment integrations
    that track referral quality, repeat purchases, and lifetime
    value---not just clicks

-   **Content analysis:** Scoring for originality, brand alignment, and
    community reception---turning subjective quality into measurable
    Work

-   **Engagement depth:** Time-on-task, completion rates, return
    frequency---distinguishing genuine participation from drive-by
    farming

This is where EmMittr becomes more than plumbing. **Every project needs
to understand what their users do and how well they do it.** EmMittr
provides that measurement AND attaches incentives to it. Two problems
solved with one integration.

### Layer 3: Agent-Native Measurement

AI agents can both perform and measure Work. A Moltbot skill doesn't
just execute a task---it can verify the task happened, assess its
quality, and report it as Work. EmMittr provides measurement skills that
agents can use:

-   **Verification skills:** Teach agents to confirm that actions
    actually occurred and met quality thresholds

-   **Scoring skills:** Agents that evaluate output quality---code
    review, content grading, data validation---and translate scores into
    Work amounts

-   **Orchestration skills:** Agents that manage multi-step workflows
    where each step is measured and rewarded independently

As AI agents become the interface for more products, measurement itself
becomes a capability you can teach. EmMittr's SDK grows with the
ecosystem.

## Part VII: Risks and Mitigations

### Volume Risk

**Risk:** Trading volume dies, fees disappear, emmissions become
worthless.

**Mitigation:** Work creates ongoing engagement. Every action generates
content/utility that can drive attention and trading. Unlike static
tokens, EmMittr projects have built-in activity engines.

### Wash Trading Attack

**Risk:** Attacker wash trades to inflate pool, capturing more value.

**Mitigation:** The fee split makes this expensive. For every \$1 wash
traded, attacker loses \$0.012 to creator + protocol (60% of 2% fee).
They'd need \>60% of emmissions to break even. The math makes it
prohibitively costly.

### Reserve Depletion

**Risk:** High Work volume depletes reserve before organic growth.

**Mitigation:** Decay curve naturally limits emmissions. Each Work
produces fewer emmission tokens. After reserve depletes, new emmissions
require direct pool contribution (price still stable).

## Appendix A: Glossary

### Clanker

A token launchpad on Base that deploys ERC-20 tokens via Farcaster.
EmMittr provides a native alternative with emmission mechanics built in.

### Decay Curve

The emmission reduction formula: Tokens(n) = Base / (1 + K × n), where n
= number of prior Work events and K = decay constant. Early participants
receive more emmissions per Work than later participants.

### Derivative Token

A token whose value is derived from an underlying asset or pool. In
EmMittr, emmissions are derivative tokens---their value derives from the
staked project tokens and accumulated LP fees in the pool.

### Emmission

A derivative token (em{Token}) representing a staked position in a
fee-earning LP pool. Minted when users perform Work, backed by project
tokens transferred from the reserve to the pool. The yield comes from LP
trading fees compounding in the pool.

### em{Token}

Emmission naming convention. For a base token called BIRD, the emmission
is emBIRD. Emmissions can be traded freely after the 30-day vesting
period.

### Emmitt

The act of triggering an emmission when Work is performed. Reserve
tokens move to pool, user receives em{Token}. The core loop: Work →
Emmitt → Earn.

The protocol and launchpad for participation-based token economics.

### EmMittrEngine

The main smart contract orchestrating EmMittr mechanics for a specific

The emmission liquidity pool. Holds staked project tokens against which

### Fee Split (40/50/10)

Distribution of trading fees: 40% compounds in EmPool (backing

The on-chain function on EmMittrEngine that projects call (via SDK) to

### Reserve

A portion of project token supply (e.g., 5%) set aside at launch to back
emmission distribution. When emmissions are minted, equivalent project
tokens transfer from reserve to EmPool, maintaining price stability.

A launch mechanism where the pool starts with 100% tokens and 0% ETH at

where using AI capabilities earns emmissions.

### Vesting Lock

A 30-day transfer restriction on newly minted emmissions. Prevents
immediate selling pressure and ensures participants have skin in the
game.

### Work

Any action that creates value for a project and triggers emmission
distribution. The project determines what counts as Work and reports it
via the EmMittr SDK. Examples: generating content (Emerge), invoking a
skill (Moltbot), quality referrals (Shopify), minting a game card,
labeling data.
